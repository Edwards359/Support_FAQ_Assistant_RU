# Интеграция ассистента в сайт или корпоративную систему

## Вариант 1: Быстрый запуск по ссылке (без разработки)

Подходит для пилота:
- публикуем GPT с режимом `Anyone with link`;
- размещаем кнопку/ссылку на сайте и во внутренних порталах.

Плюсы: быстро и дешево.  
Минусы: нет глубокого контроля UX и аналитики в вашем контуре.

## Вариант 2: Интеграция через API (рекомендуется для продакшена)

Архитектура:
1. Фронтенд-виджет чата на сайте.
2. Бэкенд-прослойка компании (Node.js/Python).
3. Вызов OpenAI API с системным промптом и базой знаний.
4. Логирование диалогов и метрик в корпоративную БД.

Что дает:
- единый стиль и брендированный интерфейс;
- контроль доступа и ролей;
- хранение истории и SLA-метрик.

## Вариант 3: Корпоративный контур (CRM/Helpdesk)

Подходит, если уже есть Zendesk/Service Desk/Bitrix/Jira Service Management:
- ассистент отвечает на первую линию;
- сложные случаи автоматически переводятся в тикет.

## Минимальный план интеграции через API

1. Определить канал: сайт, Telegram, Teams, внутренний портал.
2. Создать backend endpoint `/chat/support`.
3. Хранить API-ключ только на сервере.
4. Добавить rate limit и базовую модерацию.
5. Добавить маркер эскалации: `handoff_required=true`.
6. Настроить мониторинг: среднее время ответа, доля эскалаций, CSAT.

## Лимиты длины и расход токенов (демо `integration/website-chat`)

- Системный промпт и база знаний задают **краткие** ответы и **отказ вне темы поддержки** — см. `system-prompt.md` и `integration/website-chat/systemPrompt.js`.
- На сервере: переменные окружения `MAX_INPUT_CHARS` (по умолчанию 2000) и `MAX_OUTPUT_TOKENS` (по умолчанию 512) ограничивают длину вопроса и ответа API.
- Эндпоинт `GET /config` возвращает актуальные лимиты для виджета.
- Подробный регламент обновления файлов знаний: `deployment/knowledge-update.md`.

## Пример payload для серверной интеграции

```json
{
  "session_id": "user-123",
  "message": "Как восстановить пароль?",
  "channel": "website",
  "locale": "ru"
}
```

## Инструкция по обновлению базы знаний

См. полный регламент в **`deployment/knowledge-update.md`**. Кратко:

1. Обновите файлы в `knowledge/` (только актуальная версия).
2. Укажите дату изменения в начале документа.
3. Синхронизируйте **Инструкции** GPT с `system-prompt.md` при изменении политики.
4. Повторно загрузите файлы в Knowledge у GPT и обновите `knowledgeBase.js` при API-интеграции.
5. Запустите regression-проверку из `tests/test-cases.md`.
6. Опубликуйте новую версию и зафиксируйте changelog.
